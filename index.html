The Basics

Every value in Javascript can have a set of operations you can run on them. For example, say we have a variable named message: 

                        
                            //Accessing the operation 'toLowerCase'
                            //on 'message' and then calling it
                            message.toLowerCase();

A big difference between Javascript and Typescript is that Javascript does what's called dynamic typing: it can only provide feedback on things after the code has ran. Typescript on the other hand does what's called static typing: it is able to provide predictions about what code is expected before it runs.

A good example of this is as follows: 

                        
                            const user = {
                                name: "John",
                                age: 30
                            };

                            user.location; //returns undefined"

Here, we have a variable named user that is an object with two properties: name and age. We then try to access the property location on user. This will return undefined because location is not a property on user. This is an example of dynamic typing: we can only find out that location is not a property on user after the code has ran. In Typescript, this would show a warning prior to running the code.

Explicit Types

Typescript is able to define explicit typings for variables, parameters, etc.

                    
                        function greet(person: string, date: Date) {
                            console.log(`Hello ${person}, today is ${date.toDateString()}!`);
                        }

Here we can see the parameters person and date are of types string and Date respectively. This allows Typescript to restrict things in the future, as well as show you more information in errors.

Strictness

Typescript allows for each developer the amount of strictness that they want. If you only want to validate a small section of your code, Typescript will infer, and use the most lenient type (as well as not checking for null/undefined values). You could also validate as much as you can, and Typescript will follow that strictness.

Diving deeper into what that inference means if you aren't validating a whole lot, Typescript will pick the most lenient type when you don't define one: any. This type allows for an experience synonymous with Javascript. However, it is pointless to only use the any type, as you could just use Javascript instead.

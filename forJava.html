JavaScript

If you're coming from Java or C#, it is important to still understand a little bit about Javascript, specifically the runtime behaviors, as Typescript still runtime as Javascript.

Classes and OOP

C# and Java are mandatory OOP languages, Typescript isn't. This allows for things like:

Free Functions

Javascript functions can live anywhere, data can be passed freely without being in a pre-defined class or struct. There isn't really an implied OOP hierarchy.

Static Classes

Singletons and static classes are unnecessary in Typescript.

OOP in Typescript

If wanted, you could still work with classes and such. It is important to remember, that because of the way Typescript works, you can't query the type of any value, as they're fully erased at runetime. There are functions built in however, to help with this such as typeof and instanceof.

Rethinking Types

Typescript's types are a bit different, very useful to think of them as sets.

For example, in C# or Java, it would be very difficult to pass something around that is either a string or an int. However, if you were to think of it as belonging to the union of both the string and the number sets, it starts to become clear.

Structural Types

There doesn't have to be a specific declared relationship between stuff as well, structure plays a role here.

For example, you are able to define an instance of an interface without explicitly defining the relationship. If the structure remains the same, it is implied that they are related.

                    
                        interface Point {
                            x: number;
                            y: number;
                        }
                        function printPoint(p: Point) {
                            console.log(p.x + ", " + p.y);
                        }
                        let point = { x: 12, y: 26 };
                        printPoint(point); // Prints "12, 26"

Here we see the implied relationship between the Point interface and the variable point.
